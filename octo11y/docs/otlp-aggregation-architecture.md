# Aggregation Architecture

OTLP JSON is the single data format at every layer: input parsing, in-memory
representation, on-disk storage, aggregation, and view generation.

## Data flow

```
benchmark tool output
  → parser (parse-go, parse-rust, etc.)
  → OtlpMetricsDocument
  → stash action (writes to bench-data branch)
  → aggregate action (produces derived views)
  → chart package (renders dashboards)
```

## Design goals

1. **OTLP is the only data format.** Every component operates on
   `OtlpMetricsDocument`. Parsers produce it, actions consume it, storage
   persists it.

2. **Small, purposeful frontend fetches.** The frontend should not fetch
   hundreds of raw run files. Aggregate produces view-shaped artifacts.

3. **Preserve raw telemetry.** Raw OTLP stays available for re-aggregation
   and diagnostics.

4. **No frontend cross-file joins.** The frontend may reshape one dataset
   locally, but does not perform warehouse-style joins.

## Non-goals

- Full PromQL compatibility
- Mandatory metrics database
- DuckDB-Wasm as a default dependency

## Raw storage layout

```text
bench-data
└── data/
    ├── runs/
    │   ├── {run-id}/
    │   │   ├── benchmark.otlp.json     # benchmark results
    │   │   └── telemetry.otlp.jsonl.gz # monitor/diagnostic data
    │   └── ...
    ├── index/                           # navigation indexes
    ├── series/                          # pre-aggregated time series
    └── views/                           # UI-shaped detail views
```

Principles:

- Raw files are immutable (append-only by stash)
- Derived files are regenerated by aggregate
- Aggregate is the single writer for index, series, and view artifacts

## Semantic conventions

See [`otlp-semantic-conventions.md`](otlp-semantic-conventions.md) for the full
contract. Summary of required attributes:

**Resource:** `benchkit.run_id`, `benchkit.kind`, `benchkit.source_format`

**Datapoint:** `benchkit.scenario`, `benchkit.series`, `benchkit.metric.direction`

## Aggregation layers

### A. Navigation indexes

Small files loaded on most page visits:

- `data/index/runs.json`
- `data/index/refs.json`
- `data/index/prs.json`
- `data/index/scenarios.json`
- `data/index/metrics.json`

### B. Overview views

Files tailored to top-level product surfaces:

- `data/views/metrics/{metric}/summary.json`
- `data/views/prs/{pr}/summary.json`
- `data/views/refs/{ref}/latest.json`
- `data/views/scenarios/{scenario}/summary.json`
- `data/views/leaderboards/{metric}.json`
- `data/views/regressions/recent.json`

### C. Focused detail views

Medium-sized files loaded after user intent is clear:

- `data/views/runs/{run-id}/detail.json`
- `data/views/prs/{pr}/detail.json`
- `data/views/scenarios/{scenario}/detail.json`
- `data/views/metrics/{metric}/series.json`

### D. Raw telemetry (on demand)

- `data/runs/{run-id}/telemetry.otlp.jsonl.gz`

## Frontend responsibilities

The frontend may:

- locally reshape one already-scoped dataset
- hide/show/filter/group series inside a single detail dataset
- aggregate over attributes within one dataset

The frontend should not:

- fetch many run files and join them
- reconstruct the entire benchmark model from scratch

## Temporality

Prefer **cumulative** temporality for stored data:

- more resilient to dropped samples
- easier to re-aggregate later
- friendlier for rate/increase transforms

If delta metrics are ingested, preserve that fact for downstream interpretation.

## Custom aggregation extension point

After built-in views are stable, support a post-aggregate hook so users can
generate additional derived views:

```text
data/custom/**
```

Possible forms: JS/TS hook, Python script, DuckDB batch recipe.

## Optional advanced query

For power users, an optional DuckDB-Wasm analysis mode can provide SQL over
normalized OTLP data. This is not the default dashboard path — it has higher
bundle cost and startup overhead.
