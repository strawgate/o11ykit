+# Direct-to-storage flush prototype results (reserve-and-write)
+
+Date: 2026-04-19
+
+## Scope
+
+Prototype focus: **Approach 1 (reserve-and-write)**.
+
+- Added optional `StorageBackend.reserveBatch()` + `commitReservedBatch()` hooks.
+- Implemented hooks in `FlatStore`.
+- Updated `flushSamplesToStorage()` to use reserve/write/commit when available.
+- Kept `append`, `appendBatch`, and `parseOtlpToSamples` public signatures unchanged.
+
+I also ran a small follow-up experiment for the optional question about `pending` map iteration (`Map.values()` vs flat array iteration).
+
+## Benchmark method
+
+Workload:
+
+- Synthetic OTLP payload with 10,000 points
+- ~256 unique series (same shape as prior ingest benchmarking)
+- Node.js process on this runner
+- 120 iterations with first 20 dropped as warmup
+- Parse and flush measured separately (`parseOtlpToSamples` then `flushSamplesToStorage`)
+
+Command:
+
+```bash
+node research/ingest-experiments/02-direct-flush-bench.mjs
+```
+
+## Results
+
+### 1) End-to-end parse/flush comparison
+
+| Variant | Parse mean (ms) | Flush mean (ms) | E2E mean (ms) |
+|---|---:|---:|---:|
+| Baseline path (no reserve hooks visible to ingest) | 8.22 | 9.45 | 17.67 |
+| Reserve-and-write path (FlatStore hooks enabled) | 7.88 | 8.39 | 16.27 |
+
+Observed deltas (mean):
+
+- Flush: **-1.06 ms** (**~11.2% faster**)
+- E2E: **-1.40 ms** (**~7.9% faster**)
+
+Interpretation: reserve-and-write gives a measurable win in the flush-heavy phase and a modest but real E2E improvement.
+
+### 2) Copy-cost decomposition (where flush time goes)
+
+I split out rough costs using the same pending batches:
+
+| Micro-step | Mean (ms) |
+|---|---:|
+| Conversion only (`number[]` -> `BigInt64Array` + `Float64Array`) | 1.36 |
+| `appendBatch` insertion into store arrays | 14.08 |
+| Reserve-direct insertion loop | 14.47 |
+
+Key takeaway: in this setup, the **store insertion work dominates** over conversion-only cost. Eliminating intermediate typed-array creation helps, but it's not the entire flush bottleneck.
+
+(These micro numbers are directional only; they include per-iteration series resolution overhead and are not a drop-in replacement for the phase-level benchmark above.)
+
+### 3) Pending `Map` vs flat array iteration (follow-up)
+
+| Iteration mode | Mean (ms) |
+|---|---:|
+| `for (const batch of pending.values())` | 0.0045 |
+| `for (i=0; i<array.length; i++)` over prebuilt array | 0.0047 |
+
+Result: no meaningful win from replacing the map solely for flush iteration in this workload.
+
+## Recommendation
+
+### Worth pursuing: **Yes, with guardrails**
+
+The reserve-and-write API is worth continuing:
+
+- It is backward-compatible (optional interface extension).
+- It reduced flush mean by ~11% and E2E by ~8% in this prototype.
+- It removes one intermediate batch copy path and lets storage backends own write buffers.
+
+### Suggested next steps
+
