+# 02 — Direct-to-storage flush prototype results
+
+Date: 2026-04-19
+
+## Scope
+
+Objective: prototype direct-to-storage ingest flush paths that remove the temporary `BigInt64Array` + `Float64Array.from()` copy done in `flushSamplesToStorage`.
+
+Required context reviewed:
+- `packages/o11ytsdb/src/types.ts`
+- `packages/o11ytsdb/src/flat-store.ts`
+- `packages/o11ytsdb/src/ingest.ts`
+- `packages/o11ytsdb/src/chunked-store.ts`
+
+## Implemented prototypes
+
+### 1) Reserve-and-write (StorageBackend optional fast-path)
+
+Added an optional API to `StorageBackend`:
+- `reserveBatch?(id, count): ReservedBatch`
+- `ReservedBatch` exposes writable typed-array views (`timestamps`, `values`) plus `commit(written?)`
+
+`FlatStore` implementation:
+- Ensures growth once for the requested count.
+- Returns subarray views into the store’s owned arrays at the current write offset.
+- `commit()` advances `count` and global sample count without additional copies.
+
+`flushSamplesToStorage` now:
+- Uses `reserveBatch` if available (direct write path)
+- Falls back to old `appendBatch` behavior when unavailable
+
+Backward compatibility:
+- Existing `append`, `appendBatch`, and query API unchanged.
+- New method is optional.
+
+### 2) Streaming append prototype (append-per-sample)
+
+Prototyped an alternative flush path (`flushSamplesToStorageViaAppend`) that iterates pending samples and calls:
+- `storage.append(id, tsNs, value)` per point
+
+This avoids both temporary typed-array allocations and `appendBatch` copy, but introduces per-sample method call overhead.
+
+### 3) Pending Map → flat array exploration
+
+Added a benchmark variant that materializes `Array.from(pending.values())` once and iterates by index for the reserve-and-write path (`reserve-and-write + flat-array iteration`).
+
+## Benchmark method
+
+Runner:
+- `node research/ingest-experiments/02-direct-flush-bench.mjs`
+
+Workload:
+- 10,000 synthetic gauge points
+- ~256 unique series shape (same pattern as existing ingest benches)
+
+Measurement:
+- Parse and flush measured separately with `performance.now()`.
+- 20 warmups, 120 measured iterations.
+- Fresh storage per iteration.

+## Analysis
+
+### Which copy is expensive?
+
+Observed from deltas vs baseline:
+- Reserve-and-write removes intermediate typed-array creation + copy, but improves flush by only ~0.283ms (~3.6%).
+- Therefore, the removed copy is real but not dominant in this workload.
+- A larger remaining cost appears to be per-sample conversion (`number ms -> bigint ns`) and general iteration overhead.
+
+### Why append-per-sample did better here
+
+In this FlatStore-only prototype, append-per-sample unexpectedly won:
+- No temporary typed arrays
+- No `.set()` copy into store buffers
+- `FlatStore.append` is very cheap (bounds/grow check + direct write)
+
+This may not generalize:
+- For backends where per-sample append triggers chunk logic/compression behavior, batch paths may still win.
+- For now this is a FlatStore prototype signal, not a universal API recommendation.
+
+### Pending map vs flat array
+
+`Array.from(pending.values())` + indexed loop showed measurable gain in this run:
+- Reserve flush: 7.594ms -> 7.218ms (~5.0% faster)
+
+Interpretation:
+- Iteration overhead of Map values iterator is non-trivial at this scale.
+- A parse-time structure that already stores pending batches in a flat list (plus key->index map for lookup) is worth a focused follow-up.
+
+## Recommendation
+
+1. **Pursue reserve-and-write as an optional fast-path** (already prototype-complete).
+   - It is backward-compatible and low risk.
+   - Keep fallback to `appendBatch` for stores that do not implement reserve.
+
+2. **Follow up with pending-structure refactor** (map+array hybrid).
+   - Expected upside appears larger than copy elimination alone in this benchmark.
+   - Shape: keep map for fingerprint lookup, maintain `pendingBatches: PendingSeriesSamples[]` for flush iteration.
+
+3. **Treat append-per-sample as backend-specific optimization, not default API direction yet.**
+   - It performs best on FlatStore here.
+   - Needs validation on Chunked/Column stores before broader adoption.
+
+If we prioritize likely ROI for next iteration, do:
+- reserve-and-write (done), then
+- pending iteration structure optimization,
+- and optionally batch bigint conversion experiments (e.g., reducing BigInt creation overhead).
